Redis穿透，击穿和雪崩问题

发表于2021-03-10|更新于2021-03-10|Redis

redis缓存使用流程介绍

  • 使用redis作为缓存，客户端访问数据时，会优先访问redis，若redis没有要访问的数据，则再去访问数据库，流程如下
  1. 客户端发起请求访问数据，
  2. 访问redis
  3. 有数据，则直接返回给客户端
  4. 没有数据，则访问mysql，将数据保存到redis，并返回给客户端

缓存穿透

缓存穿透是指客户端访问一个redis种没有，且mysql中也没有的数据，大量的空数据访问会频繁的访问mysql，导致mysql缓慢或崩溃。

  • 缓存穿透问题给了黑客攻击的机会

解决方案

1.客户端过滤

在客户端发起请求时，不先访问redis，先判断该值是不是合法，比如，通常我们数据库中的主键都是大于0的数字，如果要查询小于等于0的主键，则客户端直接返回。

  • 优点：无效数据不会访问到redis，给redis减轻压力
  • 缺点：无法过滤所有请求，导致系统复杂度提高
2.缓存不存在的key

如果查询数据库也没有数据，同正常流程一样，将查询的key也保存到redis中，这样大量的无效访问只会在第一次访问到mysql，之后就会查询到redis中的数据，并返回。

  • 优点：系统复杂度低，只是正常缓存无效值即可。
  • 缺点：
    • 无效值做了缓存，意味着缓存层中存了更多的键，需要更多的内存空间（如果是攻击，问题更严重），比较好的方式时给无效值设置较短的过期时间。
    • 可能造成数据不一致问题，比如缓存了无效值之后，mysql中就有了该值，此时查询便与数据库中不一致。
3.布隆过滤器

布隆过滤器：将存在的key事先存到布隆过滤器中，在用户端访问一个key的时候，先通过布隆过滤器判断是否存在该值。若不存在，则直接返回。

  • 特性：布隆过滤器可以确定某一个值一定不存在

  • 原理：

  1. 布隆过滤器是一个字节数组，初始值都为0，当存入一个key时，会使用多个不同的hash函数对key进行hash，并将计算的hash值映射到数组的某一个位置，将该位置的值修改为1

  2. 当查询一个key是否存在时，若使用这几个hash算法得到的数组位置的值有一个为0，则表示key不存在

  3. 若hash之后数组的位置的值都为1，则表示key可能存在。

  • 优点：缓存空间占用少
  • 缺点：
    • 代码维护复杂
    • 有一定的误判率
    • 加入的key不可删除

缓存击穿

缓存击穿：当一个热点数据在redis中突然过期，大量的访问直接落到了mysql上，造成数据库压力突然增大。

  • 如何区分缓存击穿和缓存穿透？
    • 缓存击穿：击穿，本来有key，大量访问突突突的访问，突然就给打没了，就冲破了防线，落到了数据库
    • 缓存穿透：穿透，redis中露着缝隙，被钻了空子

解决方案：

  • 设置热点数据永不过期
  • 每个请求在访问redis发现没数据要访问数据库的时候，加上互斥锁，再访问数据库。

缓存雪崩

缓存雪崩： 缓存雪崩是指缓存中数据大批量同时过期（或者突然缓存不可用，重启等），导致大量请求同时访问到数据库，导致数据库访问压力突增

解决方案：

  1. 缓存数据的过期时间设置随机，防止同一时间大量数据过期现象发生。
  2. 保证redis高可用
  3. 如果缓存数据库是分布式部署，将热点数据均匀分布在不同搞得缓存数据库中。
  4. 设置热点数据永远不过期。
  5. 数据预热：重启后，使用接口或者其他手段将热点数据事先刷入redis
